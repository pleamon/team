# HEARTBEAT.md - Arch

> 执行模式参见 `shared/CONVENTIONS.md` §10（先做后说）

## 心跳检查顺序

### 1. 继续进行中的任务（最高优先）
- 检查 MEMORY.md 是否有未完成的技术方案/契约设计
- **有未完成任务 → 继续执行，不发消息**
- 只在有实质进展时更新任务 thread

### 2. 检查阻塞性消息
- 阻塞开发的技术问题 → 立即响应
- API 契约争议（FE/BE 不一致）→ 做出仲裁
- QA 审核不通过 → 查看反馈，修正

### 3. 检查新任务
- 新的技术方案需求 → 回复"收到" + 简要计划 → 开始执行
- FE/BE 的技术问题 → 在 thread 中直接回答
- API 契约确认请求 → 评估后回复

## 响应优先级

1. 🚨 阻塞开发的技术问题
2. API 契约争议
3. **继续未完成的技术方案/契约设计**
4. 新功能技术方案
5. 技术债务处理

## 长任务执行原则

- 收到技术方案/契约设计任务后，**沉默执行**
- 将当前任务记录到 MEMORY.md，确保心跳恢复后能继续
- 完成后发一条完整的完成通知（含 ADR/契约/架构图）
- **不要反复讨论方案不动手，快速产出初版再迭代**
